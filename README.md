# Ride Buds — Interactive Demo

**Live demo: [https://YOUR-DEMO-URL.vercel.app](https://YOUR-DEMO-URL.vercel.app)**

Ride Buds is a progressive web app for group motorcycle rallies in the Indian Himalayas, built solo for the Tank Up Kaza ride (Delhi → Kaza through Kinnaur). A crew joins with a shared ride code — no accounts — and gets a live map of each other's check-ins, community-flagged hazards, verified fuel stops, a pace-aware itinerary planner, group chat, and a QR-shareable emergency card. This repo is a fully clickable, zero-backend demo of the production app.

## The problem

A Himalayan group ride scatters fast. Riders leave at different times, ride at different paces, and lose mobile data for hours on the high stretches. The group needs to know roughly where everyone is, what's broken on the road ahead, and where the last fuel pump is before a 130 km gap — without anyone babysitting a tracking app or draining a battery they'll need for navigation. And if a rider goes down alone, a stranger should be able to find their blood group and emergency contacts in two taps.

## What it does

- **Live crew map** — riders drop check-in beacons (24 h expiry), hazard pins (72 h expiry) and side-trip recommendations on a shared map; every pin shows live weather, wind and elevation for that exact spot, with an AMS-altitude warning above 3,000 m
- **Route intelligence** — a 488-point road-snapped Delhi→Kaza polyline, hand-verified fuel stops with tap-to-navigate, clustered fuel/lodging POIs, and a danger-zone banner for the Pooh→Kaza no-fuel stretch
- **Pace-aware itinerary planner** — pick relaxed / standard / fast / custom km-per-day and the route segments itself into day-stages ending at real towns; start from Delhi, Chandigarh, or any city (a free-form pre-journey phase is generated automatically); saved plans appear on the rider's profile
- **Group chat** — realtime in production, with unread badges
- **Emergency layer** — an SOS button that composes a location-stamped SMS to your ICE contacts, and a public, login-free ICE card (blood group + tap-to-call contacts) designed to be opened from a QR sticker on a helmet
- **No accounts** — join with a ride code, name and optional phone; sessions are HttpOnly HMAC-signed cookies, and re-joining with the same phone re-attaches to your existing rider record

## Production stack

| Layer | Technology |
|---|---|
| Frontend | Next.js App Router, TypeScript, Tailwind v4, PWA (service worker + install hint) |
| Backend | Supabase Postgres + Realtime — service-role writes, row-scoped JWT minted per session for Realtime |
| Maps | Google Maps (`@vis.gl/react-google-maps`) + `@googlemaps/markerclusterer`; OSRM road-snapped route GeoJSON |
| Free APIs | Open-Meteo (weather + elevation), Nominatim (geocoding, proxied and session-gated) |
| Hosting | Vercel |

## Decisions worth mentioning

**Check-ins, not tracking.** Continuous live location was the obvious feature and the wrong one — it drains batteries on 12-hour riding days, dies anyway on the BSNL-only stretches past Reckong Peo, and riders didn't want surveillance. Location is shared only when you tap to check in, and pins expire on their own (check-ins 24 h, hazards 72 h), so the map stays current without a moderator.

**No accounts at a trailhead.** Nobody creates a password on a gravel shoulder at 3,500 m. Joining is ride code + name + optional phone; the server signs an HttpOnly HMAC cookie and mints a row-scoped Supabase JWT for Realtime. Phone numbers are E.164-normalized so re-joining re-attaches the same rider identity — zero login UX, no duplicates.

**The route is data, not decoration.** The Delhi→Kaza line is an OSRM road-snapped GeoJSON with stage metadata baked into its properties. The itinerary planner segments it into day-stages by pace, and a saved plan round-trips through one JSONB column using a coordinate-sentinel convention that keeps free-text pre-journey days separate from structured mountain stages on rehydration.

**Safety works offline-adjacent.** The SOS button composes an SMS — not a WhatsApp message — because SMS still goes through where mobile data doesn't. The ICE card is a public page with no login, because the person who needs it has never seen the app before.

## About this demo

Everything you can click is the real production UI and logic; what's replaced is the infrastructure:

- **Supabase is stripped** — all state lives in browser memory; Realtime is simulated (wait ~45 s and a crew member checks in)
- **Google Maps → Leaflet + OpenStreetMap** — so the demo runs with zero API keys, zero env vars, and no billing account behind a portfolio piece
- **The crew is fictional** — riders, phone numbers and messages are generated by a seeded PRNG relative to today's date, so the data is stable per day and always fresh
- **Open-Meteo stays real** — pin weather and elevation are live, because that API needs no key

The production app served a real rally; its repository stays private. This demo's source is what you're reading.
